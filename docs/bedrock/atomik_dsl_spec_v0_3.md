# atomik DSL — Specification v0.3.1 (consolidated draft)

Status: consolidated draft. v0.3.1 folds the errata fed back by the render-core spec and the shipped `cycle` kernel (C1–C4) into the normative sections; the surface language is unchanged and files still begin `atomik 0.3` (changelog: §14). Produced from corpus batches 01–04. Not frozen. Every design decision below is traceable to a named rupture (A–R expressivity, G1–G6 generability, Ped-1–5 pedagogy). Section 11 is the full traceability table.

---

## 1. Purpose and design stance

atomik is a text DSL for **claim-bearing, teachable scenes**: small visual explanations that are stored as files, versioned, diffed, patched, and generated by inexpensive models.

The convergence target is a triplet, in this order of priority when in conflict:

1. **Epistemic fidelity** — a scene must never look more certain than its sources; falsehoods must be representable *as marked falsehoods*.
2. **Generation determinism** — the same input should produce stable, diffable output; ambiguity is moved out of the small model's hands.
3. **Structural + choreographic expressivity** — the recurring cognitive skeletons of explanation (structures) and the recurring teaching gestures (choreography), with a deliberately closed vocabulary.

North-star acceptance test: **the misconception case** (elicit → confront → resolve, with the false belief visibly marked false). It intersects the hardest problem of each axis. A version of atomik that cannot encode it is a diagram tool, not a learning tool. Section 9.1 proves v0.3 passes.

Non-goals for v0.3: function plotting, arbitrary programmability, pixel-level control, standalone-app interactivity. See §12 (deferred).

## 2. Ground rules (syntax-wide)

- **Line-oriented.** One statement per line. Each line is independently parseable and independently repairable. Blank lines ignored.
- **Comments**: `#` to end of line.
- **Version pragma**: first line of a block is `atomik 0.3` — the *surface* version, unchanged at spec v0.3.1 (§14).
- **Identifiers**: `[A-Za-z_][A-Za-z0-9_-]*`. Reserved id: `claim` (addressable endpoint, §5.3).
- **Strings**: double quotes, `\"` and `\\` escapes. Full Unicode; accents, apostrophes, CJK, RTL text are plain content.
- **Wikilinks**: `[[Page name]]` — a reference into the note graph. In generated scenes, only emitted when resolved (§10).
- **Uniform attributes**: any statement may end with zero or more `[key]` or `[key value]` or `[key "quoted value"]` pairs. Order-free. **Unknown attribute keys are a warning, then ignored** — this is the forward-compatibility valve.
- **Unknown top-level keywords**: warning + line skipped; the rest of the scene still parses and renders (graceful partial validity).
- **Degradation invariant**: an unrendered scene must read as a sensible outline in any plain editor. This is a testable requirement on the syntax, not a hope.
- **Canonical form**: a normalizing printer orders statements (pragma, scene, claim, subject, group, node/evidence in first-mention order, relation, place, data, project, input, derive, rule, step, mark) and normalizes whitespace. Generators MUST emit canonical form. This is a diff-stability requirement (G1).

## 3. The four planes and the keyword inventory

16 top-level keywords across four planes, plus 6 effect words that appear only inside `rule`/`step`.

| Plane | Keywords | Role |
|---|---|---|
| **Frame** (epistemic envelope) | `atomik` `scene` `claim` `subject` | what the scene asserts, about what, with what status |
| **Model** (content) | `node` `evidence` `relation` `group` `place` `data` | the knowledge structure — no visuals |
| **Projection** (rendering choice) | `project` | which archetype renders the model; flippable |
| **Reactive + choreography** | `input` `derive` `rule` `step` `mark` | learner inputs, computed values, conditions, authored reveal sequence |
| Effects (inside rule/step; `set`/`require` step-only — C2) | `note` `reveal` `hide` `highlight` `set` `require` | bounded verbs |

The model plane is the spine. Everything else annotates or projects it. A minimal valid scene is `scene <id>` + `claim "<text>"` + at least one model statement.

Renames vs v0.2: `state` → `input` (H: collision with state machines, the most common CS diagram); `layout` → `project as` (A: projection is a first-class, flippable choice, not a styling afterthought).

## 4. Grammar (EBNF, per line)

```ebnf
file      = pragma? line* ;
pragma    = "atomik" VERSION EOL ;
line      = ( scene | claim | subject | node | evidence | relation
            | group | place | data | project | input | derive
            | rule | step | mark | COMMENT | BLANK ) EOL ;

attrs     = ( "[" KEY ( VALUE | STRING )? "]" )* ;

scene     = "scene" ID attrs ;
claim     = "claim" STRING attrs ;                     (* status attr: see §5.1 *)
subject   = "subject" ( WIKILINK | STRING ) attrs ;

node      = "node" ID ( STRING | WIKILINK ) attrs ;
evidence  = "evidence" ID STRING attrs ;               (* sugar: node with role=evidence *)
relation  = "relation" ( ID ":" )? end ( "->" | "~" ) end KIND? attrs ;
end       = ID | "claim" ;                             (* KIND is a free token *)
group     = "group" ID STRING? attrs ;
place     = "place" ID ( "at" SCALAR | RELPOS ID ) attrs ;
relpos    = "above" | "below" | "left-of" | "right-of"
          | "inside" | "adjacent" ;
data      = "data" ID ( "cols" | "row" ) CELL ( "|" CELL )* ;

project   = "project" "as" ARCHETYPE ( "from" ID )? RANGE? attrs ;

input     = "input" ID "=" ( "slider" RANGE
                           | "choice" STRING ( "|" STRING )*
                           | "toggle" ) attrs ;
derive    = "derive" ID "=" expr ;
rule      = "rule" expr "=>" rule-effect ;             (* C2: set/require step-only *)
step      = "step" INT step-effect ;
rule-effect = "note" STRING | "reveal" ID+ | "hide" ID+
            | "highlight" ID+ ;
step-effect = rule-effect | "set" ID SCALAR | "require" ID ;
mark      = "mark" "meter" STRING "value" ID attrs ;

expr      = or ; or = and ( "or" and )* ; and = cmp ( "and" cmp )* ;
cmp       = sum ( ("=="|"!="|"<"|"<="|">"|">=") sum )? ;
sum       = term ( ("+"|"-") term )* ; term = fact ( ("*"|"/") fact )* ;
fact      = SCALAR | STRING | ID | "not" fact | "(" expr ")" ;
```

The expression grammar is deliberately this small and closed. There is no user-defined function, no loop, no string manipulation. That is the "no arbitrary script" invariant, kept.

The effect grammar splits by host (v0.3.1 — C2): `set` writes an input, and a rule that fires on input change could re-trigger itself, so `set` is step-only (applied once, on step entry); `require` declares a step's gate — a property of the step, not a reactive effect — and is step-only too. Either inside a `rule` is a validator **error, in both profiles**.

## 5. Closed vocabularies

### 5.1 Epistemic status — `[status …]` on claim, node, evidence, relation

`established · supported · contested · hypothesis · speculative · reported · misconception · unspecified`

- `reported` = "asserted by the source, not verified here." **The safe default for generated scenes** — it never upgrades a source's hedging.
- `misconception` = held-but-false. Valid on nodes/evidence (the belief being taught against), not on the scene claim.
- `unspecified` = the default in hand-authored scenes when omitted.

### 5.2 Epistemic class of a relation — `[as …]`

`fact · inference · hypothesis · analogy · interpretation · refutation · boundary`

The relation KIND stays a free token (`causes`, `supports`, `maps-to`, `refutes`, `supersedes`…) so authors keep domain language; `[as …]` is the closed class the validator and renderer understand. A `maps-to [as analogy]` renders differently from a `causes [as fact]`, and a `[as refutation]` renders as an *undoing* arrow, not an asserting one.

### 5.3 Addressable relations and the `claim` endpoint

`relation r1: iau2006 -> pre2006 supersedes` gives the edge an id. Ids let relations target relations (argument maps: an objection to an *inference*; analogy boundaries: "breaks *here*") and let evidence target the scene claim: `relation iau2006 -> claim supports`. This one small mechanism carries the whole evidence layer (P) and the boundary gesture (Ped-3).

### 5.4 Node roles — `[role …]`

`process (default) · decision · start · terminal · question · evidence · assumption · contradiction`

Roles are structural semantics (a decision diamond, a proof assumption), not colors. `tone` remains available for mood only.

### 5.5 Common attributes (all optional)

`status · as · role · in <group> · sign +|- · weight <0..1> · many · label "<text>" · source <wikilink|string> · date <value> · tone · salience criterial|incidental · default · max · scale linear|log · suggested · origin authored|generated · kind cluster|lane|loop · polarity reinforcing|balancing`

Unknown keys: warn + ignore (§2).

### 5.6 Projections — `project as …`

Closed set, each with a dedicated, tuned layout engine. **Same model, flippable projection** — the projection line is metadata, not content.

`graph (default) · flow · cycle · tree · nested · concentric · timeline · axis · matrix · bar · map`

- `timeline` and `axis` consume `place <id> at <value>` (dates, magnitudes); `axis` accepts `RANGE` and `[scale log]`.
- `matrix` and `bar` consume `from <dataset>`.
- `map` consumes relative `place` constraints (`above`, `left-of`, `adjacent`…) — bounded spatial knowledge without coordinates.
- Reserved, not in v0.3: `chart` (function plotting), `venn`, `gantt` (§12).

### 5.7 Inputs and effects

Input types: `slider <range>` · `choice "a" | "b" | …` · `toggle`. An input is **committed** once the learner has interacted with it. Sliders, toggles, and a `choice` with `[default]` count as committed from the start; a bare `choice` starts uncommitted — that is what makes prediction gates bite (C4, §6).
Effects: `note` (transient message — C3, §6) · `reveal` / `hide` (structural visibility; every `reveal` target starts hidden — C1, §6) · `highlight` (emphasis, the "active element") · `set` (write an input; step-only, applied once on step entry — C2, §4) · `require` (declare a step's gate; step-only — the step's own effects wait for commitment and advancing past it is blocked — C4, §6).

## 6. Semantics that need stating

**Initial visibility (C1).** Every id named as a target of any `reveal` effect — in a `step` **or** a `rule` — starts hidden; everything else is visible from step 1. An id targeted by both `reveal` and `hide` starts hidden. The flag is computed per element by lang-core (render-core D4); a rule's reveal would be meaningless if its target began visible.

**Steps.** The learner navigates steps in order (UI: next/prev; backward navigation is always free and never un-commits an input). Step effects are cumulative: at step *n*, the effects of steps 1…*n* have applied. Steps are **authored sequence** (the teaching choreography); `input` is **learner-driven**. The two are distinct and may coexist.

**Gated steps (C4).** A step with one or more `require <input>` lines is a *gated step*. Its **own effects apply only while every required input is committed** — `step 2 require guess` + `step 2 reveal vacuum` keeps the evidence hidden until the learner has predicted (predict-then-see, the north-star shape) — and advancing past it is blocked until then (the UI shows the lock and names the input). Commitment is defined in §5.7. Export/print renders the all-revealed state, gates ignored, with step badges on step-revealed elements: a printed scene must not silently hide teaching content.

**Rules.** `rule <expr> => <effect>` re-evaluates whenever an input or derived value changes. Rules may `note/reveal/hide/highlight` — this is how learner actions can alter structure (prediction gates, branch on wrong guess). Rule effects are **reactive, not latched**: they hold while the condition holds and release when it stops; presentation stays a pure function of `(currentStep, inputs, committed)`. `set` and `require` are illegal inside a rule — validator error, in both profiles (C2, §4).

**Notes (C3).** A `note` is transient: the visible notes are the current step's notes plus the notes of currently-true rules, nothing older. A note is voice-over for the moment, not a persistent annotation on the scene.

**Groups.** `group <id> "<label>"` + membership via `[in <id>]` on nodes. `[kind lane]` for side-by-side regions (the two halves of an analogy), `[kind loop] [polarity reinforcing]` for causal-loop labeling.

**Place.** `place x at 7.4` positions by value on `axis`/`timeline` (B). `place heart above diaphragm` is a relative constraint consumed by `map` (R) — best-effort constraint solving, author never writes coordinates.

**Data.** `data <id> cols …` then `data <id> row …` lines; self-contained lines (repair-friendly) rather than an indented block. Rendered by `matrix`/`bar`.

**Evidence.** `evidence <id> "<text>" [source …] [date …] [status …]` is sugar for a node with `role evidence`. Evidence↔evidence relations (`contradicts`, `supersedes`) and evidence→`claim` relations are the disputed-fact machinery.

## 7. Rendering invariants (renderer contract)

1. **Marked falsehood invariant**: any element with `status misconception` MUST be rendered visibly negated (struck, flagged, badged "false belief") in every projection and every step in which it is visible. A misconception may never render neutrally. (Ped-2, G3.)
2. **Status is visible**: the claim's status renders as a chip on the scene. `reported` and `contested` must be visually distinct from `established`.
3. **Refutation is visually an undoing**: `[as refutation]` and `[as boundary]` edges must not look like assertion arrows.
4. **Suggested projection is flippable**: a `project … [suggested]` line renders with an affordance to switch archetype; flipping rewrites only that line.
5. **Polish ≠ truth**: renderers must not add certainty theater (checkmarks, authoritative styling) beyond what statuses state. Ties into Truth Lens.

## 8. Generation pipeline contract (co-designed, not optional)

The DSL is not evaluated in isolation from its generator (batch-03 conclusion). Four stages:

1. **Frame** — determine scene scope from the highlight (default: one claim per scene; the user can widen/narrow the selection). (G5)
2. **Ground** — resolve candidate `[[links]]` against the real note index *before* generation. Unresolved → plain string literal, never a fabricated wikilink. Link candidates may be offered in UI, not in the file. (G2)
3. **Generate, strict profile** — the generator emits **model plane only**, in canonical form, plus at most one `project as … [suggested]` line. Claim `[status …]` is REQUIRED; the value may not be stronger than the source's own commitment (hedged source ⇒ `hypothesis`/`contested`/`reported`, never `established`). Every relation extracted from prose defaults `[as inference]` unless the source states it as settled fact. (G1, G3, G4)
4. **Validate + repair** — diagnostics are line-scoped and machine-actionable (`L12: unknown-id 'glucos' — nearest 'glucose'`); the repair loop patches single lines. (G6)

**Validation profiles**: `authored` (lenient: statuses optional, defaults to `unspecified`) vs `generated` (strict: the rules above are errors, not warnings). The profile is declared via `scene … [origin …]` or by the pipeline.

## 9. Worked examples (proof against the corpus)

### 9.1 North star — the misconception case (corpus P1)

```atomik
atomik 0.3
scene falling_objects [origin authored]
claim "In the absence of air, all objects fall at the same rate." [status established]
subject [[Free fall]]

node belief "Heavy objects fall faster" [status misconception]
node vacuum "Hammer and feather dropped in a vacuum" [role evidence]
node truth "Both land together" [role terminal]
node reason "Air resistance, not mass, caused the difference"

relation vacuum -> belief refutes [as refutation]
relation vacuum -> truth shows [as fact]
relation reason -> truth explains [as inference]

input guess = choice "hammer first" | "together" | "feather first" [label "Predict:"]

step 1 reveal belief
step 1 note "Most people expect the heavy object to win."
step 2 require guess
step 2 reveal vacuum
step 3 reveal truth
step 3 highlight truth
rule guess == "together" => note "You predicted it — now here's why."
rule guess != "together" => note "A very common intuition. Watch what actually happens."
step 4 reveal reason
```

Elicit (step 1, marked-false belief) → commit (gate) → confront (evidence, refutation edge) → resolve (reason). Every element that was inexpressible in batch 04 is now a first-class line. This is the acceptance test, passed on paper; §13 says how to test it for real.

### 9.2 Disputed fact with sources (corpus §17, the epistemic hard case)

```atomik
atomik 0.3
scene planet_count [origin generated]
claim "The Solar System has eight planets." [status contested]
subject [[Planet]]

evidence iau2006 "IAU 2006: eight planets; Pluto reclassified" [source [[IAU 2006 resolution]]] [date 2006]
evidence pre2006 "Pre-2006 consensus: nine planets" [date 1930]

relation iau2006 -> claim supports [as fact]
relation pre2006 -> claim contradicts [as fact]
relation r1: iau2006 -> pre2006 supersedes [as interpretation]

project as flow [suggested]
```

N sources on one claim, evidence-vs-evidence, claim as endpoint, statuses that keep the dispute visible. (P, E, G4.)

### 9.3 Analogy with a boundary (corpus P4)

```atomik
node coin "Spinning coin"
node qubit "Qubit in superposition"
relation map1: coin ~ qubit maps-to [as analogy]
node ent "Entanglement — no coin counterpart"
relation ent -> map1 breaks [as boundary]
```

The lesson *is* the boundary, and the boundary now has grammar. (Ped-3.)

### 9.4 Comparison matrix (corpus §7)

```atomik
data protocols cols "Protocol" | "Connection" | "Reliability" | "Speed"
data protocols row "TCP" | "connection-oriented" | "reliable" | "slower"
data protocols row "UDP" | "connectionless" | "best-effort" | "faster"
project as matrix from protocols
```

(I, D2 — categorical data minimal; function plotting stays out, §12.)

### 9.5 Value-positioned axis (corpus §9)

```atomik
node lemon "Lemon juice"
node blood "Blood"
place lemon at 2
place blood at 7.4
project as axis 0..14 [scale log] [label "pH"]
```

(B, and the same `place` keyword covers relative anatomy: `place heart above diaphragm` + `project as map` — R.)

## 10. What the generator may and may not do (summary card)

MUST: emit canonical form; model plane only (+ one `[suggested]` projection); a `status` on the claim, never stronger than the source; `[as inference]` default on extracted edges; literals for unresolved references.
MUST NOT: fabricate wikilinks; upgrade hedged→established; choose the projection silently; emit steps/choreography unless the user asked for a teaching sequence; strip stated uncertainty, evidence, anomalies, or alternatives when they are in the highlighted text.

## 12. Deferred / reserved (explicitly out of v0.3.x)

`chart`/`plot` (function curves, tangents, intersections — a second engine; delegate or defer) · `repeat` (parametric/recursive expansion — tension with no-script) · `frame` (small multiples with cross-frame identity) · `venn` · `gantt` · rich cardinalities · relation-to-relation layout beyond midpoint attachment · style theming beyond `tone`.

Each is reserved as a keyword so future versions don't break v0.3 files (unknown-keyword rule §2 gives old renderers graceful skip).

## 13. What would falsify v0.3.1 (the tests that matter)

1. **Corpus re-encode**: re-write all 20+ batch cases in v0.3; every previously-inexpressible line must now be expressible or explicitly deferred in §12. (Paper test — can be done now.)
2. **Generability eval** (from batch 03): real small models, logged highlights, multiple runs; score archetype stability (should approach 100% under model-only generation), reference accuracy, epistemic fidelity, confabulation rate, one-pass repairability. Compare model-chooses-projection vs model-emits-model-only regimes. — **Executed at v0.3.1 (CP-DSL-003): Haiku 4.5 + Gemini 3.1 Flash-Lite, 16 tasks × 2 regimes × 5 runs, kernel-as-grader + Sonnet 5 confabulation judge. Findings: `docs/evals/generability_eval_v0_3_1.md`. Headline — 0% fabrication both models, misconception-preservation 100%/95% (both pass), status-ceiling hold 91%/84%, archetype stability ~84–85%; shared weaknesses (high judged confabulation, low cross-run structural stability) confirm the risk is in generation *judgment*, not the language. The model-only regime (R1) lifts archetype stability over the free regime (R2), most on Haiku.**
3. **Pocket-spec budget**: the model-facing spec must fit the founding constraint (≈≤2K tokens). Measured in the companion file. — **Holds at v0.3.1 (~1.2K tokens; CP-DSL-001). The eval used the pocket spec verbatim as the system prompt.**
4. **Misconception acceptance**: a strict-profile generation over a misconception-teaching passage must yield §9.1-grade output (marked falsehood preserved, no flattening) in ≥ an agreed threshold of runs. — **Executed (CP-DSL-003), threshold ≥80%: Haiku 100% (20/20), Gemini 95% (19/20) of R1 runs on misconception tasks preserved the marked falsehood. Both PASS.**
5. **Degradation read-aloud**: unrendered v0.3 files given to a reader cold; if they can't summarize the scene, the outline invariant fails.

## 14. Changelog

### v0.3 → v0.3.1 (2026-07-07, consolidation — CP-DSL-001)

Writing the render-core spec and shipping the `cycle` kernel fed four errata back into the language (render-core §2.3 D4/D11 and §5; C4 was found while writing the golden fixture's runtime oracle). v0.3.1 folds them into the normative sections. **The surface language is unchanged**: files still begin `atomik 0.3`, no keyword or vocabulary moved, and the Scene IR (irVersion 0.1) and golden fixture are untouched. One validation rule is tightened (C2).

- **C1 — initial visibility generalized** (§6). Any target of any `reveal` — step *or* rule — starts hidden; a reveal+hide conflict resolves to hidden. v0.3 said only "named in some `step n reveal`", which made a rule's reveal meaningless.
- **C2 — `set` illegal in rules** (§3, §4, §5.7, §6). Validator **error in both profiles**: a rule that writes an input can loop; runtime purity (render-core D11) forbids it. In a `step`, `set` stays legal and applies once, on entry. The split effect grammar additionally records what the kernel always enforced: `require` (a gate declaration, not a reactive effect) is step-only too.
- **C3 — note lifetime defined** (§5.7, §6). Notes are transient: the current step's notes plus currently-true rules' notes; nothing persists across steps.
- **C4 — gated-step semantics** (§5.7, §6). A step with `require` withholds its **own effects** until every required input is committed, and blocks advancing. Replaces v0.3's weaker "blocks advancing past *n*" reading, under which a same-step reveal could spoil the prediction. Commitment-by-default is spelled out: sliders, toggles, and `choice` with `[default]` yes; bare `choice` no.
